# Спецификация: валидатор каталога eBay (`ebay_validation_catalog`)

## 1. Назначение

Сервис непрерывно проверяет объявления (items), попавшие в каталог eBay-парсера, и
складывает вердикты в собственную базу. Одобренные объявления — вход для следующего
этапа конвейера (детальная обработка item, база `ebay_validation_item`).

Проверки: состояние товара, блоклист/вайтлист слов в названии, максимальная цена
с учётом доставки, дедупликация по продавцу и по названию. Валидатор не парсит eBay
сам — он только читает чужие базы и пишет в свою. Единственное исключение: при смене
названия объявления он ставит задачу на перепарсинг для парсера.

## 2. Окружение и базы

Все базы — Postgres-контейнеры на сервере `194.164.245.107`, в docker-сети `db_default`.
Изнутри сети контейнеры видны по имени (порт 5432), снаружи — по IP и проброшенному порту.
Учётка везде `admin` / пароль в `.env`.

| Роль | База (контейнер) | Внешний порт | Доступ |
|---|---|---|---|
| Источник каталога и объявлений | `ebay_data` | 5415 | чтение + NOTIFY-триггеры |
| Максимальные цены закупки | `parts_prices` | 5416 | только чтение |
| Маппинг артикулов на смарт-артикулы | `parts_research` | 5410 | только чтение |
| База валидатора (вердикты, курсоры, задачи) | `ebay_validation_catalog` | 5421 | своя, читает-пишет |

Что именно читается:

- `ebay_data.catalog_items` — присутствие item в каталоге: `article`, `context_id`,
  `item_id`, `first_seen_at`, `last_seen_at`, `misses`, `is_active`. Артикул (`article`) —
  поисковый запрос каталога. FK на `items`: строка каталога всегда имеет строку item,
  но item может быть «заглушкой» (все данные NULL, пока парсер не обработал).
- `ebay_data.items` — данные объявления: `title`, `condition` (нормализован парсером до
  `'new'` / `'other'`), `price_usd`, `seller_id`, `is_dead`, `first_seen_at`.
- `ebay_data.item_shipping` — стоимость доставки per (item, context): `shipping_cost`, `updated_at`.
- `ebay_data.changes` — журнал изменений, который ведёт парсер: `changed_at`, `item_id`,
  `field_id` (словарь в `fields`; title = scope `item`, name `title`), `kind`, `old_value`, `new_value`.
- `parts_research.smart.part_articles` — `article → part_id`: какие артикулы относятся
  к какому смарт-артикулу. Один смарт-артикул = одна запчасть, у неё несколько артикулов.
- `parts_prices.buying.smart_prices` — `smart_part_id`, `max_buy_price_usd` (уже в USD),
  `updated_at`. Максимальная цена закупки смарт-артикула; может быть NULL (не задана).

Единица валидации — тройка **(item_id, part_id, context_id)**: один и тот же item
оценивается в рамках смарт-артикула, к которому привели его артикулы каталога.
В `catalog_items` членство детальное — item, найденный по пяти артикулам одной
смарт-детали, имеет пять строк; валидатор агрегирует их в одну запись, а полный список
артикулов хранит в колонке `articles` (пересобирается при каждой ревалидации). Item,
найденный по артикулам двух разных смартов, — две независимые записи. Сейчас работаем
с одним zip (одним контекстом), но контекст везде входит в ключи, чтобы добавление
контекстов ничего не ломало.

## 3. Проверки

Состав и порядок проверок задаются списком `checks` в конфиге. Проверки выполняются
по порядку с обрывом на первой провалившейся (short-circuit): в `reject_reasons`
попадает одна причина — первая. Проверки, отсутствующей в списке, не существует
(так они выключаются). Объявление одобрено, если дошло до конца списка.

Перед списком всегда выполняется одна встроенная, невыключаемая проверка:

- **`inactive`** — объявление мёртвое (`items.is_dead`) или выпало из каталога
  (`catalog_items.is_active = false`) → отказ. Мёртвые объявления не участвуют ни в
  каких проверках, включая соревнование дедупа.

Конфигурируемые проверки:

- **`condition`** — `items.condition` должен входить в список `allowed_conditions`
  из конфига. По умолчанию `["new"]`.
- **`blocklist`** — в названии встречается слово из блоклиста → отказ.
- **`whitelist`** — название не содержит ни одного слова из вайтлиста → отказ.
  Пустой вайтлист = проверка выключена.
- **`price`** — `price_usd + shipping_cost > max_buy_price_usd` смарт-артикула → отказ.
  Доставка неизвестна (нет строки в `item_shipping`) → считается 0.
  `max_buy_price_usd` равен NULL → ценовая проверка пропускается (объявление идёт дальше).
- **`dedup`** — дедупликация по продавцу и по нормализованному названию внутри
  смарт-артикула, обе одной проверкой. Это не одиночная проверка, а соревнование:
  участвуют живые объявления смарт-артикула, **прошедшие проверки, стоящие в списке
  до `dedup`**. Победитель на каждый слот (продавец; название) — самое раннее по
  `items.first_seen_at`. Проигравшие получают отказ `seller_dup` / `title_dup`,
  победители продолжают проверки, стоящие после `dedup`.

Позиция `dedup` в списке — содержательный выбор. `dedup` первым: «берём первое
объявление продавца, и только его проверяем дальше» — победитель определяется чистой
хронологией, и если он не проходит дальнейшие проверки, у продавца в этом смарте нет
одобренных. `dedup` последним: победитель выбирается только среди полностью годных
объявлений, плохие в соревновании не участвуют.

Слот держит текущий победитель соревнования. Если он умирает или перестаёт проходить
пред-дедупные проверки — состав соревнования меняется, группа смарт-артикула
пересматривается в том же тике, слот занимает следующий по `first_seen_at`.

Сопоставление слов (блок/вайтлист): без учёта регистра; `match_type: word` — совпадение
целого слова (не внутри другого слова), `match_type: substring` — вхождение подстроки.

Нормализация названия для дедупа: нижний регистр, все последовательности символов,
кроме латинских букв и цифр, схлопываются в один пробел, края обрезаются.

## 4. Статусы и жизненный цикл

Статус записи — одно из трёх:

- **`pending`** — item есть в каталоге, но парсер ещё не заполнил данные (`title` NULL).
  Проверки не проводились. Когда парсер заполнит item, событие придёт само (см. §5).
- **`approved`** — все проверки пройдены.
- **`rejected`** — есть причины отказа (список в `reject_reasons`).

Отзыв одобрения фиксируется **отдельно от статуса**, колонкой `revoked_at`: она
проставляется в момент перехода `approved → rejected` и сбрасывается в NULL при
возврате в `approved`. Так потребитель, уже взявший item в работу, гарантированно
видит факт отзыва, сколько бы прогонов ни прошло после него.

Изменение правил конфига (блок/вайтлист и пр.) **не** вызывает переоценку уже
проверенного — правила применяются только к новым и изменившимся объявлениям.

### Перепарсинг при смене названия

Если в журнале `changes` появилось изменение поля `title`, валидатор:
1. ставит задачу в свою таблицу `reparse_tasks` (если активной задачи по этому item ещё нет);
2. сразу ревалидирует item по текущим данным, не дожидаясь перепарса.

Парсер заберёт задачу, перезапишет данные item — это породит новые события, и item
ревалидируется ещё раз уже по свежим данным. Другие изменения (цена, доставка, смерть)
задач на перепарс не порождают — только ревалидацию.

## 5. Как валидатор узнаёт о работе

Три механизма, от дешёвого к дорогому:

### 5.1 Отпечаток (fingerprint) — «надо ли вообще пересматривать»

Для каждой проверенной записи хранится md5-отпечаток всех входов вердикта:
`title | seller_id | condition | price_usd | shipping_cost | max_buy_price_usd | alive`,
где `alive = is_active AND NOT is_dead`. Кандидат с совпавшим отпечатком пропускается
без валидации (обновляется только `last_checked_at`). Любое значимое изменение — включая
смену максимальной цены смарт-артикула и смерть объявления — меняет отпечаток и
автоматически ведёт к ревалидации.

Отпечаток не включает групповое состояние (кто держит слот дедупа): пересмотр группы
запускается явно, когда меняется состав соревнования (см. §3), — таким записям
отпечаточный скип не применяется.

### 5.2 Горячий цикл — курсоры + NOTIFY

Раз в `tick_interval_sec` (и немедленно при получении NOTIFY) валидатор собирает
кандидатов инкрементальными запросами «что появилось после закладки» — закладки
(курсоры) хранятся в таблице `cursors` и двигаются только после успешной обработки тика:

| Курсор | Запрос | Что ловит |
|---|---|---|
| `items_first_seen` | `items.first_seen_at > pos` | новые объявления (в т.ч. дозаполнение заглушек не ловит — это ловит `changes`) |
| `catalog_first_seen` | `catalog_items.first_seen_at > pos` | новые строки каталога |
| `changes` | `changes.changed_at > pos` | любые изменения объявлений, включая смерть; здесь же отлавливается смена title для `reparse_tasks` |
| `shipping_updated` | `item_shipping.updated_at > pos` | появление/изменение доставки |
| `smart_prices_updated` | `buying.smart_prices.updated_at > pos` | смена максимальных цен — кандидатами становятся все объявления смарт-артикула |

Каждый запрос делается с перекрытием `cursor_overlap_sec` назад от закладки — защита от
транзакций, закоммиченных «в прошлое». Повторная обработка безвредна: валидация
идемпотентна, а совпавший отпечаток превращает повтор в бесплатный скип.

На старте курсоры стоят в начале эпохи — первый тик естественно прогоняет весь каталог.

NOTIFY — только сигнал «проснись сейчас», его потеря ничего не ломает (тик всё равно
случится по таймеру). В `ebay_data` создаются statement-level триггеры (функция
`validator_notify()`, канал `validator_events`): на INSERT в `items`, `catalog_items`,
`changes` и на INSERT/UPDATE в `item_shipping`. Триггеры пассивные — данные не трогают.
Мелкая известная дыра: вставка напрямую в партицию `changes_YYYYMM` мимо родителя
statement-триггер не зажжёт — это покрывается обычным тиком по таймеру.

Маппинг артикулов (~2 тыс. строк) и цены смартов (~1 тыс. строк) загружаются в память
целиком в начале каждого тика — это дешевле любого кэширования.

### 5.3 Полная сверка — страховка

Раз в `full_reconcile_interval_sec` (по умолчанию сутки; момент последней сверки
переживает рестарт — хранится в `cursors` под именем `last_reconcile_at`) валидатор
проходит весь каталог чанками, считает отпечатки фактического состояния и сравнивает
со своими. Несовпавшие — ревалидируются. Это ловит всё, что могло пройти мимо журнала
и курсоров (ручные правки, сбои, дыру с партициями). Замерено на сервере: ~16 с и
~50 МБ трафика на 2 млн строк — приемлемо для суточной периодичности.

Вместе со сверкой чистятся выполненные задачи `reparse_tasks` старше
`reparse_done_retention_days` дней.

## 6. Схема базы валидатора

База `ebay_validation_catalog`, схема `public`. Миграции — SQL-файлы в
`migrations/validator/`, применяются при старте сервиса (учёт — в `schema_migrations`).

### `validated_items` — вердикты (главная таблица-результат)

| Колонка | Тип | Описание |
|---|---|---|
| `item_id` | bigint | id объявления eBay |
| `part_id` | text | смарт-артикул (`smart_…`) |
| `context_id` | smallint | контекст (marketplace+zip) из `ebay_data.contexts` |
| `articles` | text[] | все артикулы смарт-артикула, под которыми item сейчас числится в каталоге (пересобирается при ревалидации) |
| `seller_id` | integer | продавец на момент проверки |
| `title` | text | название на момент проверки |
| `title_norm` | text | нормализованное название (ключ дедупа) |
| `condition` | text | состояние на момент проверки |
| `price_usd` | numeric | цена на момент проверки |
| `shipping_cost` | numeric | доставка на момент проверки (NULL = не было данных) |
| `max_buy_price_usd` | numeric | максимальная цена смарта на момент проверки |
| `status` | text | `pending` / `approved` / `rejected` |
| `reject_reasons` | text[] | причина отказа — первая провалившаяся проверка (§3); пустой массив у approved/pending |
| `fingerprint` | bytea | md5-отпечаток входов вердикта (§5.1), NULL у pending |
| `revoked_at` | timestamptz | момент отзыва одобрения (§4), NULL если не отзывалось/возвращено |
| `first_validated_at` | timestamptz | когда запись впервые создана |
| `validated_at` | timestamptz | последнее изменение вердикта или данных снапшота |
| `last_checked_at` | timestamptz | последний раз, когда запись рассматривалась (включая скип по отпечатку) |

Первичный ключ `(item_id, part_id, context_id)`. Индексы: `(part_id, seller_id)`,
`(part_id, title_norm)`, `(status)`, `(validated_at)`.

**Контракт потребителя:** одобренное — `status = 'approved'`; отозванное —
`revoked_at IS NOT NULL`. Инкрементальное чтение — курсором по `validated_at`
(индекс есть): при любом изменении вердикта или снапшота поле обновляется, скипы его
не трогают.

### `reparse_tasks` — задачи парсеру

| Колонка | Тип | Описание |
|---|---|---|
| `task_id` | bigserial PK | |
| `item_id` | bigint | что перепарсить |
| `reason` | text | причина (`title_changed`) |
| `created_at` | timestamptz | когда поставлена |
| `taken_at` | timestamptz | когда парсер взял (ставит парсер) |
| `done_at` | timestamptz | когда выполнена (ставит парсер) |

Частичный индекс по `item_id where done_at is null` — валидатор не создаёт дубль,
пока по item висит активная задача.

**Контракт парсера:** подключиться к базе валидатора, забирать пачкой
`UPDATE … SET taken_at = now() WHERE task_id IN (SELECT … WHERE taken_at IS NULL ORDER BY task_id LIMIT N FOR UPDATE SKIP LOCKED) RETURNING …`,
после перепарса проставить `done_at`. Валидатор сам узнает о новых данных через `changes`.

### `cursors` — закладки

| Колонка | Тип |
|---|---|
| `name` | text PK |
| `pos` | timestamptz |
| `updated_at` | timestamptz |

### `schema_migrations` — учёт применённых миграций (`name`, `applied_at`).

### Объекты в `ebay_data` (миграция `migrations/ebay_data/`)

Функция `validator_notify()` и четыре statement-level триггера `validator_notify_*`
(см. §5.2). Миграция идемпотентна (`create or replace` / `drop trigger if exists`)
и применяется при каждом старте сервиса.

## 7. Конфигурация

Два источника, не пересекаются:

- **`.env`** — только подключения (секреты, не коммитится): `EBAY_DATA_DSN`,
  `PARTS_PRICES_DSN`, `PARTS_RESEARCH_DSN`, `VALIDATOR_DSN`. Локально — IP и внешние
  порты, в проде — имена контейнеров и 5432 (см. `.env.example`).
- **`config.yaml`** — правила и параметры (коммитится). Читается **один раз при
  старте**; изменил — перезапусти сервис (`docker compose restart`).

```yaml
tick_interval_sec: 30            # период горячего цикла
cursor_overlap_sec: 60           # перекрытие курсорных запросов назад
full_reconcile_interval_sec: 86400  # период полной сверки
reparse_done_retention_days: 7   # хранение выполненных задач перепарса

allowed_conditions: [new]        # допустимые items.condition

checks:                          # состав и порядок проверок; нет в списке = выключена;
  - dedup                        # обрыв на первой провалившейся; позиция dedup задаёт,
  - condition                    # кто участвует в соревновании (прошедшие проверки до него, §3):
  - blocklist                    # dedup первым = «берём первое объявление продавца,
  - whitelist                    # и только его проверяем дальше»
  - price

rules:
  blocklist:                     # слово найдено -> отказ
    - pattern: fake
      match_type: word           # word | substring
      scope: global              # пока только global; задел: smart:<id> | brand:<x> | category:<x>
      active: true
  whitelist:                     # ни одно слово не найдено -> отказ; пустой список = выключено
    require: any                 # any = достаточно одного совпадения
    words:
      - pattern: impeller
        match_type: word
        scope: global
        active: true
```

Поле `scope` сейчас принимает только `global`; иные значения зарезервированы и
отвергаются при старте — формат файла при появлении скоупов не изменится.
