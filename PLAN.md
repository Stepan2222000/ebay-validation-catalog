# План реализации валидатора каталога

Спека: [SPEC.md](SPEC.md). Каждый этап завершается терминальной проверкой;
к следующему переходим только после зелёного результата предыдущего.

## Этап 1. Каркас репозитория

**Делаем:**
- `git init`; `.gitignore` (`.env` в git не попадает — SPEC §7);
- `config.yaml` ровно по примеру из SPEC §7 (список `checks`, правила, интервалы);
- `.env.example` с двумя вариантами DSN: локальные IP:порт и прод-имена контейнеров
  из таблицы SPEC §2;
- структура папок: `src/validator/`, `migrations/validator/`, `migrations/ebay_data/`, `tests/`;
- `pyproject.toml` (зависимости: asyncpg, pyyaml).

**Не делаем:** ни строчки логики, никаких подключений к базам, ничего на сервере.

**Проверка:** конфиг парсится, дерево репозитория чистое.

## Этап 2. Миграции обеих баз

**Делаем:**
- `migrations/validator/001_init.sql` — таблицы и индексы по колонкам из SPEC §6:
  `validated_items` (с `articles text[]`, `revoked_at`), `reparse_tasks`, `cursors`;
- `migrations/ebay_data/001_notify.sql` — функция `validator_notify()` и 4 statement-level
  триггера из SPEC §5.2, идемпотентно (накатывается при каждом старте — SPEC §6);
- применение к реальным базам; повторное применение как тест идемпотентности.

**Не делаем:** существующие таблицы `ebay_data` не изменяем (триггеры пассивные);
сервисный код не пишем.

**Проверка:** инспекция созданных объектов в обеих базах; повторный накат без ошибок.

## Этап 3. Ядро валидации + e2e-тест на синтетике

**Делаем:**
- загрузка и валидация конфига (`scope` кроме `global` отвергается при старте — SPEC §7);
- нормализация названия и отпечаток с `alive` внутри (SPEC §5.1);
- встроенная проверка `inactive` (SPEC §3);
- одиночные проверки и порядок `checks` с обрывом на первой провалившейся (SPEC §3);
- соревнование `dedup`: участники — прошедшие проверки до него по списку, победитель
  по `first_seen_at`, семантика позиции в списке (SPEC §3);
- переходы статусов `pending/approved/rejected` и `revoked_at` (SPEC §4);
- агрегация членства в `articles[]` (SPEC §2);
- e2e-тест в транзакциях с откатом на реальных серверах: все сценарии согласованного
  прототипа (дедуп по продавцу/названию, блок/вайтлист word и substring, цена с
  доставкой, condition, pending → approved, смерть победителя → переезд слота, смена
  макс. цены, NULL-цена, скип по отпечатку) + сценарии «dedup первым против последним».

**Не делаем:** курсоры, NOTIFY, циклы — ядро тестируется на подсунутых данных.

**Проверка:** e2e зелёный; обе базы после прогона нетронуты.

## Этап 4. Горячий цикл (тик, курсоры, NOTIFY, reparse)

**Делаем:**
- тик по SPEC §5.2: пять курсорных выборок с перекрытием `cursor_overlap_sec`,
  загрузка маппинга артикулов и цен смартов в память;
- постановка `reparse_tasks` при смене title без дублей активных задач
  (SPEC §4, контракт парсера — SPEC §6);
- инвариант «курсоры двигаются только после успешной обработки тика» (SPEC §5.2);
- LISTEN канала `validator_events` как немедленный триггер тика (SPEC §5.2);
- главный цикл сервиса, graceful shutdown.

**Не делаем:** полную сверку; деплой.

**Проверка:** запуск против реальных (пустых) баз — мигрирует, слушает, тикает
вхолостую; живой тест: синтетическая вставка в транзакции → NOTIFY будит → тик
подбирает кандидатов (с откатом).

## Этап 5. Полная сверка + чистка задач

**Делаем:**
- суточная сверка чанками со сравнением отпечатков, `last_reconcile_at` в `cursors`
  (SPEC §5.3);
- чистка выполненных `reparse_tasks` старше `reparse_done_retention_days` (SPEC §5.3).

**Не делаем:** оптимизации сверки сверх замеренного (~16 с на 2 млн строк — приемлемо
для суточной периодичности, SPEC §5.3).

**Проверка:** прогон на синтетике с искусственно испорченной записью — сверка находит
и чинит; откат.
